## v2.3.2 — 修原生壳里弹窗(confirm/alert/prompt)点了没反应

### 修复(影响面很大)
- **原生 App 里所有需要弹窗确认的操作点了都没反应**(对比「↻ 重启后端 / ＋ 新对话」、删除对话、重命名、更新、删 MCP …):根因是原生壳的 WKWebView **没实现 JS 对话框代理(WKUIDelegate)**,导致系统 `confirm()` 直接返回 false、`alert()`/`prompt()` 静默失效——网页版正常、原生版全哑。两路都修:
  - **原生壳补上对话框代理**:`confirm`/`alert`/`prompt` 用原生 NSAlert 弹出。**需重新安装 App 才生效**(原生壳不随界面在线更新)。
  - **界面自带页内对话框兜底**:不依赖原生弹窗——`alert()` 改为页内 toast、关键确认(对比重启/新对话)改为页内确认框。**这条随界面在线更新即可生效,不必重装**。

## v2.1.9 — 联网引导改为自动探测(只在被劫持的机器上发)

### 修复
- **v2.1.8 的「走 curl/MCP」引导原来是无条件发的** —— 只对挂 fake-ip 代理的机器成立,普通机器 `fetch_url` 本来正常,却被误导去用 curl。现 server.py 启动时**实测本机 DNS**:把公网域名解析到保留/私有地址(如 `198.18.x.x`)才判定为 fake-ip 环境(新增 `/api/netenv`,结果缓存 120s)。前端只在 `fakeip=true` 时才发那句引导。

  - **fake-ip 机器(Clash/Surge/MacPacket 等):自动开启**引导,无需手动配置。

## v2.1.8 — 对比模式联网取数引导(本机代理下)

### 改进
- **对比模式开 Shell 时,引导各模型走 curl / MCP 取网页全文**:本机若挂 fake-ip 代理(域名解析成 198.18.x.x 保留地址),CodeWhale 的内置 `fetch_url` 抓全文会被其硬编码的 SSRF 守卫拦(报 `resolved IP ... restricted address`)——这是 CodeWhale 上游的安全限制,`[network].allow` 也盖不住(已实测)。现开 Shell 时附一句指引,让模型直接用 `exec_shell` + `curl`(或 MCP 的 fetch / searxng)读全文,**跳过会被拦的 fetch_url、不再先弹红字再回退**。`web_search`(搜索)本就不受影响、照常可用。指令只进 prompt 不进界面气泡。
  - 注:彻底的原生修法是把本机代理从 fake-ip 切到 redir-host 模式(DNS 返回真实 IP),内置 web 工具即全局可用,无需引导。

## v2.1.7 — 切换对话提速 + 修发送 409

### 修复 / 提速
- **重任务跑满后端时切换对话不再卡顿**:之前一条对话在跑(联网/工具/后台 shell)会占满后端,叠加每 4 秒一次的侧栏轮询**无防重入**(慢请求层层堆叠)→ 切换像死机、任务一结束才恢复。现侧栏轮询**防重入**(上一发没回来不再压新的),聚合超时 20s→8s(忙时快速失败不占连接)。
- **切换到「非默认模型」对话不再连环卡**:`ensure_provider_server` 原来在等 app-server 就绪(最长 ~20s)的全程持锁,启动 A 会阻塞切到 B。现**就绪等待移出锁**,各 provider 后端并行启动互不阻塞;并发请求同一冷后端**只启一个进程**(启动锁去重)。
- **热切换近乎零延迟**:provider 后端「活着」状态缓存 15s,省去每次请求都探活的 HTTP 往返。
- **修「发送失败 HTTP 409 Thread already has an active turn」**:切回一条仍在跑的对话时,快照不会重放已发生的 turn → 界面误以为空闲、再发就 409。现切换时按 `turns` 末轮状态**同步"运行中"**(发送自动转排队);万一仍撞 409 也**优雅转排队**(上一轮完成后自动发),不再弹红字报错。
