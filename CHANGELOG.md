## v2.5.1 — 中文 OCR 修复 + 补丁二进制自动刷新

### 修复
- **图片中文 OCR 一塌糊涂 → 修好**:`image_ocr`(macOS Vision)之前没设识别语言,默认基本只认英文,中文图全乱甚至被当成"清晰度不够"。补丁加 `recognitionLanguages = [zh-Hans, zh-Hant, en-US]` + 抓更小的字。实测中文发票/单据从全乱码 → 近乎全对。

### 机制
- **补丁二进制版本感知自动刷新**:server.py 启动时比对签名 manifest 里的二进制 SHA,变了就自动重下(只下变化的那个)。所以这种二进制层升级,旧机器在线更新到 2.5.1 后会**自动**拿到,无需重装。

## v2.4.5 — 对比支持附件/粘贴 + 修单窗口实时消息不显示 + 首条消息提速

### 修复(重要)
- **新建对话发第一条消息要等好几秒才显示**:① `send()` 里 `await loadThreads()`(刷侧栏列表)会阻塞发送,而该聚合查 :7878 summary 随会话数变慢(50 条 ~4.5s)→ 首条消息要等它跑完才出现。改成**不阻塞**(后台刷侧栏)。② 单窗口原来不"乐观渲染"——发出去要等后端起 turn 经 SSE 才显示;现在**发出去立刻显示你的消息 + 进入运行态**(真事件到达时认领该气泡,不重复)。实测首条消息 5.6s → ~0.4s。③ `/api/threads/all` 加 6s 缓存(命中后侧栏秒刷),失败回退上次结果。
  - 注:推理模型(如 deepseek-v4-pro)首个**回答**仍需思考几秒(过程在折叠块),这是模型本身,非界面问题。
- **主窗口发消息后对话区空白 / 加附件像没反应**:根因是新建对话(默认 provider)被建在 per-provider 后端,但后续 turns/events 被 `_route_base` 路由到主后端 :7878 ——「建会话的后端」≠「跑 turn/发事件的后端」,跨后端 seq 对不上 → 实时 SSE 只收到 `thread.started`、收不到后续 turn 事件 → **消息不显示**(历史快照因 thread 存储共享仍能加载,极具迷惑性)。这是上版修「新建对话 502」后才暴露的(之前根本建不成会话)。
  - 修:默认 provider 的新会话**直接建在主后端 :7878**(与路由一致、不再 pin);非默认 provider 才建在 per-provider 后端 + pin。实测单窗口发消息(含带附件)实时渲染恢复正常。

### 新增
- **多模型对比窗口可加附件了**:输入框旁新增 📎 按钮,可上传文件(文本/代码),也支持**直接粘贴(⌘V)剪贴板里的文件/图片** 和**拖拽文件**到对比视图。
  - 附件上传后在输入框上方显示 chip(可 ✕ 移除);发送时把文件路径折进 prompt,**发给所有选中的模型**——各栏的 agent 用 `read_file` 读同一份文件(共享文件系统路径),实现"同一份文件、多模型并排解读"。
  - 用户气泡只显示原文 + `📎 文件名`(不暴露长路径);附件发送后自动清空。
  - 实测:上传 → 两栏(DeepSeek / GPT)都 `read_file` 读到文件并答对。
  - 注:纯文本粘贴照常;图片会上传但当前文本模型看不了(同单窗口)。

## v2.4.3 — 修「在线更新按钮在旧原生壳里点了没反应」(confirm 静默失效)

### 修复
- **另一台机器点「更新到 X」没任何反应**:根因是更新前会弹一个**原生 `confirm()`** 确认框,而 v2.3.2 之前安装的旧原生壳没实现 WKWebView 的 `WKUIDelegate` → `confirm()` 直接静默返回 `false` → 更新被挡在第一步,看起来"点了没反应"(同类问题还会影响:删除对话、重命名、删 MCP、更新后端)。

- **已卡住机器的即时解法**(它收不到这个修复,因为更新动作本身被旧壳挡住):在那台机器终端跑一次 `curl -X POST "http://127.0.0.1:3000/api/update/gui/apply?token=$(cat ~/.codewhale-gui/token)" -d '{}'` 直接绕过 confirm 触发更新到本版;此后页内 `cwConfirm` 生效,应用内更新按钮恢复正常。或重装最新安装包(同时带上新原生壳)。

## v2.4.2 — 修对比「多轮发送顺序错乱 / 输入无缓存」

### 修复
- **一轮回答还没显示完就再发下一句,两栏的对话顺序会乱成「问1 问2 答1 答2」**(应为「问1 答1 问2 答2」):根因是 `cmpSend`(发给所有栏)**没有重入保护** —— 发送按钮虽 `disabled`,但 **Enter 键直接调 `cmpSend()` 绕过禁用**,于是上一轮还在派发时第二轮就并发起来,两条用户气泡被前置堆叠到答案之前。
  - **发送串行化 + 输入排队缓存**:一轮在进行中(busy / 任一栏在跑)时再发,新问题**缓存进队列**,当前轮全部完成后**自动按序发出**,保证每栏严格「问→答→问→答」。发送区显示「发送中…/排队 N」,输入框提示「回答进行中,再发会排队」。
  - **每栏追问(`cmpRunOne`)加同步派发锁**:关闭 `ensure` 网络往返期间重复 Enter 的二次进入窗口;被拦的文字保留在输入框不丢失。
  - `cmpRun` 异常分支补回 `running` 复位,防某栏卡在"运行中"导致后续发送一直排队。
  - 实测(真实前端):两轮快速连发→第二轮被排队(按钮显「排队1」、用户气泡不再前置堆叠)→ 首轮完成后自动续发→最终两栏均为「问1 答1 问2 答2」;每栏追问重复触发被锁拦截,无重复气泡。

## v2.4.1 — 修对比追问「输出与显示不同步 / ✓0s 吞答案」

### 修复
- **追问(继续某栏对话)时,栏头瞬间显示 `✓ 0s`,答案却不流式显示,要再发一句才把剩余文字逼出来**:根因是续轮的 SSE `since_seq` 用了可能过期的本地游标,把**上一轮的 `turn.completed` 事件重放**了进来 → cmpRun 立刻误判整轮结束、关掉流,新一轮的流式增量丢失。两路根治:
  - `since_seq` 改为**发送前实时取该会话当前 `latest_seq`**,只听本轮新事件、绝不重放历史。
  - 每轮**锚定自己的 `turn_id`**,SSE 只认本轮事件,重放的旧轮事件一律忽略。
  - 实测:追问栏头正常走 `思考中·0s → 输出中·3s → ✓3s`(不再 ✓0s),答案实时流式、完整显示。

## v2.1.7 — 切换对话提速 + 修发送 409

### 修复 / 提速
- **重任务跑满后端时切换对话不再卡顿**:之前一条对话在跑(联网/工具/后台 shell)会占满后端,叠加每 4 秒一次的侧栏轮询**无防重入**(慢请求层层堆叠)→ 切换像死机、任务一结束才恢复。现侧栏轮询**防重入**(上一发没回来不再压新的),聚合超时 20s→8s(忙时快速失败不占连接)。
- **切换到「非默认模型」对话不再连环卡**:`ensure_provider_server` 原来在等 app-server 就绪(最长 ~20s)的全程持锁,启动 A 会阻塞切到 B。现**就绪等待移出锁**,各 provider 后端并行启动互不阻塞;并发请求同一冷后端**只启一个进程**(启动锁去重)。
- **热切换近乎零延迟**:provider 后端「活着」状态缓存 15s,省去每次请求都探活的 HTTP 往返。
- **修「发送失败 HTTP 409 Thread already has an active turn」**:切回一条仍在跑的对话时,快照不会重放已发生的 turn → 界面误以为空闲、再发就 409。现切换时按 `turns` 末轮状态**同步"运行中"**(发送自动转排队);万一仍撞 409 也**优雅转排队**(上一轮完成后自动发),不再弹红字报错。
